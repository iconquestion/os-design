# 实验7现有代码分析与整体修改思路

本文基于当前仓库中的 `lab6` 实现进行分析，说明现有代码为何还无法完成实验7“地址空间扩展、多进程、`Exec/Exit/Join`、空闲帧管理”等目标，并给出建议的整体修改思路。

## 1. 当前代码的能力边界

从现有实现看，系统已经具备以下基础能力：

- 能通过 `StartProcess()` 装入一个用户程序并运行它，见 `lab6/progtest.cc:23-45`。
- 能在系统调用入口中处理 `Halt`、`Exit`、`Exec`、`Write`、`Yield`，见 `lab6/exception.cc:167-228`。
- 线程切换时会保存/恢复用户寄存器，并在切换回线程时恢复其地址空间页表，见 `threads/scheduler.cc:90-135`。

但这些能力仍然停留在“单地址空间模型稍作扩展”的层面，离实验7要求的“可正确支持多个用户进程并管理其生命周期”还有明显差距。

## 2. 为何现有代码无法完成实验目标

### 2.1 地址空间仍然按“单进程独占物理内存”设计

`AddrSpace::AddrSpace()` 里最核心的问题有三处，见 `lab6/addrspace.cc:60-118`：

- 页表建立时直接令 `virtualPage == physicalPage`，见 `lab6/addrspace.cc:85-97`。
- 清零内存时直接 `bzero(machine->mainMemory, size)`，见 `lab6/addrspace.cc:99-101`。
- 装入代码和数据时，直接按虚拟地址写入 `machine->mainMemory[virtualAddr]`，见 `lab6/addrspace.cc:103-116`。

这意味着当前实现默认：

- 一个进程的第 0 页一定放到物理第 0 帧；
- 一个进程的第 1 页一定放到物理第 1 帧；
- 装载新进程时会从物理内存起始处重新覆盖；
- 程序装入阶段根本没有“为该进程挑选空闲物理帧”这一步。

因此，只要有第二个用户进程被装入，新进程就会覆盖旧进程的数据，这正是实验文档里指出的问题。

### 2.2 没有全局空闲帧管理机制

实验要求使用 `Bitmap` 管理空闲帧，但当前 `AddrSpace` 类中只有：

- `TranslationEntry *pageTable`
- `unsigned int numPages`

见 `lab6/addrspace.h:21-40`。

当前代码里没有：

- 全局帧位图；
- 帧分配接口；
- 帧释放接口；
- “当前剩余物理页是否足够”这一层判断。

`lab6/addrspace.cc:77-80` 只有 `ASSERT(numPages <= NumPhysPages)`，它检查的是“单个进程大小是否不超过物理页总数”，不是“此时系统还剩多少空闲帧”。这无法支持多个进程并存。

### 2.3 地址空间析构没有回收物理帧

析构函数目前只做了：

```cpp
delete[] pageTable;
```

见 `lab6/addrspace.cc:125-128`。

它没有回收：

- 该进程占用的物理帧；
- 与该进程相关的进程号、退出状态、同步对象等资源。

因此，即便后续补上帧分配，只要析构时不释放，系统也无法长期运行多个进程。

### 2.4 `Exec()` 只“创建线程”，没有建立可管理的进程对象

当前 `SC_Exec` 的逻辑见 `lab6/exception.cc:182-203`：

1. 从用户内存复制文件名；
2. 创建一个新的内核线程 `new Thread("user process")`；
3. `Fork(StartUserProcess, filename)`；
4. 返回 `nextSpaceId++`。

这里最大的问题不是“没开出线程”，而是“返回值和真实进程之间没有建立可靠关系”：

- `nextSpaceId` 只是一个递增整数，见 `lab6/exception.cc:30`；
- 这个整数没有存进 `Thread` 或 `AddrSpace`；
- 系统里没有“pid -> 线程/进程控制块”的表；
- 父进程无法依据这个 `SpaceId` 找到子进程；
- `Join(pid)` 因而没有实现基础。

也就是说，当前 `Exec()` 返回的是“编号”，不是“可被跟踪管理的进程标识”。

### 2.5 没有 `Join()` 所需的进程生命周期信息

虽然 `syscall.h` 里定义了 `SC_Join` 与 `Join(SpaceId id)`，见 `lab6/syscall.h:20-25`、`lab6/syscall.h:55-65`，但 `ExceptionHandler()` 里没有 `SC_Join` 分支，见 `lab6/exception.cc:173-224`。

这不只是“少写了一个 case”，还意味着当前系统缺少 `Join()` 依赖的所有状态：

- 没有 pid 到线程/进程对象的映射；
- 没有父子关系记录；
- 没有退出状态保存位置；
- 没有“子进程结束后唤醒父进程”的同步原语；
- 没有防止重复 `Join` 的状态位。

因此，`Join()` 当前不是“未完成”，而是“缺少实现前提”。

### 2.6 `Exit()` 只结束当前线程，没有完成进程善后

当前 `DoExit()` 见 `lab6/exception.cc:95-116`，流程是：

1. 打印退出信息；
2. `currentThread->space = NULL`；
3. `delete space`；
4. 若还有就绪线程则 `Finish()`，否则 `Halt()`。

它能结束“当前承载用户程序的线程”，但还没有完成实验7所要求的进程退出工作，例如：

- 保存退出码供 `Join()` 返回；
- 标记该 pid 已退出；
- 唤醒等待该 pid 的父进程；
- 从全局进程表中清理该进程；
- 回收该进程全部物理页帧。

所以现在的 `Exit()` 更像“线程结束”，不是完整的“进程退出”。

### 2.7 线程结构里没有可支撑多进程管理的字段

当前 `Thread` 中与用户程序相关的字段只有：

- `int userRegisters[NumTotalRegs]`
- `AddrSpace *space`

见 `threads/thread.h:119-131`。

这意味着线程对象里还没有：

- 进程 pid；
- 父进程 pid；
- 退出码；
- 是否已退出；
- 是否已被 `Join`；
- 用于 `Join` 的信号量、锁或条件变量。

实验文档要求“为实现 `Join(pid)`，考虑如何在该进程相关联的核心线程中保存进程号”，而当前代码正缺这一层。

### 2.8 当前装载流程仍以“单程序启动”为中心

`StartProcess()` 的入口设计是一次性打开、装入、运行一个程序，见 `lab6/progtest.cc:23-45`。`StartUserProcess()` 只是把这套单程序流程搬到了新线程中，见 `lab6/exception.cc:118-141`。

这说明现有代码虽然能“再开一个线程跑用户程序”，但整体模型仍然是：

- 以线程为中心；
- 以立即运行新程序为中心；
- 而不是以“创建一个可被父进程观察、等待、回收的进程”为中心。

这也是为什么实验7必须补一层真正的进程管理逻辑。

## 3. 现阶段的主要制约点

如果把问题收敛成几个最关键的制约点，主要有以下四类：

### 3.1 内存分配模型不对

这是最底层的制约点。只要还保留“虚页号等于物理页号”和“从 `mainMemory` 起始处直接装载”的方式，多进程就一定互相覆盖。

### 3.2 缺少进程标识与进程表

没有可追踪的 pid，没有进程表，`Exec/Join/Exit` 无法闭环。

### 3.3 缺少退出同步机制

没有退出状态保存和等待/唤醒机制，`Join()` 无从实现。

### 3.4 资源回收链条不完整

即使暂时能运行多个进程，只要退出时不回收帧、不清理进程状态，系统很快就会失效。

## 4. 建议的整体修改思路

建议按“先打通内存，再打通进程，再补系统调用”的顺序改，不要一开始就直接写 `Join()`。

### 4.1 第一步：先把 `AddrSpace` 改成真正的按帧分配

目标是让每个进程的每个虚页，都能映射到“当前系统里某个空闲物理帧”。

建议做法：

1. 在 `addrspace.cc` / `addrspace.h` 中引入全局 `BitMap`，大小为 `NumPhysPages`，用于管理物理帧。
2. 在创建地址空间时，先判断空闲帧是否足够。
3. 为每个虚页从位图中分配一个物理帧，填入 `pageTable[i].physicalPage`。
4. 清零时不要再 `bzero(machine->mainMemory, size)`，而是按“该页对应的物理帧”逐页清零。
5. 装入代码和数据时，不要直接按虚拟地址写入 `mainMemory`，而要根据页表把内容拷贝到实际物理帧位置。
6. 在析构函数里遍历页表，把本进程占用的物理帧在位图中释放。

这一层完成后，多个用户进程才有可能真正共存。

### 4.2 第二步：建立最小可用的进程管理信息

目标是让 `Exec()` 返回的 `SpaceId` 真的对应一个可追踪进程。

建议至少补齐以下信息：

- 为每个进程分配唯一 pid；
- 在线程对象中保存 pid；
- 保存 parent pid；
- 建一个全局进程表，支持通过 pid 查找对应控制块。

实现形式有两种：

- 轻量做法：直接在 `Thread` 上加字段，再用一个全局表管理 `pid -> Thread*`。
- 更清晰的做法：单独定义一个 `ProcessControlBlock`，线程里只挂指针。

如果目标是先完成实验，第一种做法实现成本更低。

### 4.3 第三步：补齐 `Exec()` 的“创建进程”语义

当前 `Exec()` 里线程已经能 fork 出来，改造重点应放在“建档”上：

1. 父进程调用 `Exec()` 时先分配 pid；
2. 创建子线程；
3. 将 pid、父 pid、初始状态写入线程或 PCB；
4. 放入全局进程表；
5. 再 `Fork(StartUserProcess, ...)`；
6. 返回这个 pid。

这样返回值才真正有意义。

### 4.4 第四步：补齐 `Exit()` 的完整退出流程

建议将 `DoExit()` 扩展成真正的进程退出逻辑：

1. 保存退出码；
2. 标记进程状态为已退出；
3. 唤醒正在 `Join()` 等待它的父进程；
4. 释放地址空间和物理帧；
5. 视设计决定何时把该进程从全局表移除。

这里要注意一个设计问题：

- 如果子进程一退出就从全局表彻底删掉，父进程之后可能无法 `Join()`；
- 因此通常要保留“僵尸态”信息，至少保留到父进程取走退出码为止。

即便实验不要求完整 UNIX 语义，也建议保留“退出后仍可被 Join 一次”的最小机制。

### 4.5 第五步：实现 `Join(pid)`

在有了 pid、进程表、退出状态之后，再实现 `Join()` 就顺了。

建议逻辑：

1. 检查 pid 是否存在；
2. 检查该 pid 是否为当前进程的子进程；
3. 若子进程尚未退出，则阻塞当前线程等待；
4. 子进程退出后返回其退出码；
5. 处理重复 `Join` 的情况。

同步手段上，最简单的是给每个进程准备一个 `Semaphore(0)` 或者等价机制，子进程退出时 `V()`，父进程 `Join()` 时 `P()`。

### 4.6 第六步：补充失败路径和边界情况

建议把以下情况都统一返回错误值，而不是直接 `ASSERT`：

- `Exec()` 找不到文件；
- 新程序所需页数超过当前空闲帧；
- pid 不存在；
- `Join()` 目标不是自己的子进程；
- 同一子进程被重复 `Join()`。

这样测试会更稳定，也更符合系统调用的行为预期。

## 5. 推荐的修改顺序

为了降低调试难度，建议按下面顺序推进：

1. 改 `AddrSpace`，实现基于 `Bitmap` 的物理帧分配与释放。
2. 验证两个用户程序能否先后创建且不互相覆盖。
3. 在线程中加入 pid / parent pid / exit status 等字段。
4. 建立全局进程表。
5. 改 `Exec()`，让返回值与真实子进程绑定。
6. 改 `Exit()`，保存退出码并回收资源。
7. 实现 `Join()`。
8. 最后再补 `Print()`、错误处理和扩展系统调用。

## 6. 一句话总结

当前代码之所以无法完成实验7，不是因为“只差几个系统调用分支”，而是因为它的底层仍是“单进程独占物理内存”的装载模型，同时缺少 pid、进程表、退出状态和等待机制。要完成实验7，必须把“地址空间分配”和“进程生命周期管理”两条线一起补齐。
